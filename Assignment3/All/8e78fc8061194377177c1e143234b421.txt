				<!-- RSPEAK_START --><p>A little over a week ago, I wrote about how the authentication model for an <a href="http://docs.timdorr.apiary.io">unpublished Tesla REST API</a> was <a href="http://programming.oreilly.com/2013/08/tesla-model-s-rest-api-authentication-flaws.html">architecturally flawed</a> because it failed to take basic precautions against the sharing of credentials with third-parties common to most REST-based services these days. Since its publication, the main criticism of the article centered around the fact that the API is neither a published API nor has it been advertised as being meant for third-party consumption.</p>
<p>The adding of value to devices and services with or without the knowledge/permission of their creators is an integral part of the Internet of Things. These days, people expect an API around their devices. They will discover any APIs and add value to the device/service&mdash;even if the task requires a little reverse engineering work. A responsible creator of a device or service in today&#8217;s world defined by the Internet of Things must therefore do the following things&mdash;always:</p>
<ol>
<li>Give it a public API</li>
<li>Protect any internal communications so they can&#8217;t be reverse engineered</li>
<li>Protect any public communications so that they don&#8217;t put end users at risk when they leverage third-party devices and services</li>
</ol>
<div id='ctx_linker' class='ctx_widget'></div><span id="more-63228"></span>							
				<p>Whether the API is published or not, this perspective doesn&#8217;t consider that we&#8217;ve slowly been evolving the way we use technology. As we rethink many of the devices we use at work and home, we have to rethink our expectations not only of how these devices work in isolation, but also of how they work together. Whether it&#8217;s a phone, a car, or a light bulb, technology users should expect that these devices are capable of secure interactions.</p>
<p><strong>The Burden of Security</strong></p>
<p>The flaw I exposed in the Tesla API results from Tesla placing the burden of managing authentication credentials entirely on the shoulders of end users. In other words, the soundness of the Tesla API rests on the premise that end users should never, in any circumstances, leverage third-party tools to interact with their cars. Tesla has no responsibility to make an unpublished API secure against third-party malfeasance or misfeasance.  </p>
<p>As a general rule, I don&#8217;t like approaches to software architecture that force users to choose between leveraging useful functionality and being secure, especially when the option to have both is easily addressed. If you place the burden of security on the user instead of the API, the proper response for a user is to forgo all value-added functionality from third parties in order to operate securely. In my experience, convenience always wins in the battle of convenience versus security, and it&#8217;s folly to expect users to change.</p>
<p><strong>Three phenomena exemplify my view of this problem:</strong></p>
<ul>
<li>Cloud computing, where end users are sticking things &#8220;into the cloud&#8221; for greater agility while central IT admonishes them for &#8220;going rogue&#8221;</li>
<li>BYOD (bring your own device), where end users are leveraging a wider variety of devices than those vetted by corporate IT to enable them to do things corporate IT can&#8217;t envision</li>
<li>The &#8220;Internet of Things,&#8221; where all services and devices are interconnected via rich APIs and function well beyond the intentions of their designers</li>
</ul>
<p>If we didn&#8217;t have users going beyond the intent of the creators of a device or service, we wouldn&#8217;t have any of these things. We wouldn&#8217;t even have many of the things we take for granted today like online banking. </p>
<p>Cloud computing is most secure when the cloud providers empower users with the tools to operate securely in the cloud. Trying to stop people from going into the cloud isn&#8217;t the solution. The same goes with BYOD. You can try to stop people from using their devices on your corporate network. The result, however, will be that they will stop using your corporate network. That neither provides you with greater security nor does it stop users from using their devices to do work.</p>
<p>When speaking of the Tesla Model S, however, we&#8217;re talking about the &#8220;Internet of Things&#8221;&mdash;the point at which all devices and all services communicate with one another as needed without much, if any, human intervention. </p>
<p><strong>A Story of an Interconnected World</strong></p>
<p>One of the fundamental aspects of the Internet of Things is that the vendor doesn&#8217;t get to determine valid &#8220;use cases&#8221; for their devices or services. In fact, the point of an interconnected world is to create value well beyond the imagination of a device&#8217;s original inventors. Tesla, for example, may or may not have envisioned a world in which my car, my alarm clock, the weather, light bulbs, and my home security system are talking to each other to get me out of the house. I have and likely so have many, many others.</p>
<p>It takes only a few modern devices and some programming know-how to start building your own Internet of Things. No fancy wiring or custom hardware is necessary.</p>
<p><strong>Here are the components:</strong></p>
<ul>
<li>Light bulbs with an API like the <a href="https://www.meethue.com">Philips Hue light bulbs</a></li>
<li><a href="http://www.raspberrypi.org">A Raspberry Pi</a> or even a general purpose laptop, desktop, or server</li>
<li>A car with an API like a <a href="http://teslamotors.com">Tesla Model S</a></li>
<li>A programmable thermostat with an API</li>
<li>An internet connection</li>
<li>A tablet device</li>
</ul>
<p>The critical features of all of these components are that they can connect to the Internet and they have an API. Except for the custom programming on the Raspberry Pi, they don&#8217;t know anything about the other things they will work with.</p>
<p>They are specifically not connected through some purpose-specific local connection. None of them are built knowing anything about the others, yet they end up being fundamentally interconnected by RESTful APIs and a bit of programming.</p>
<p><strong>The Myth of the Private API</strong></p>
<p>Perhaps the root of the question where security lies boils down to the question of whether the Tesla should be considered part of the Internet of Things. While it&#8217;s cool to have your house automatically warm up your car when you wake up, it may not be a scenario in which Tesla has chosen to participate. They have not published their API and, for all we know, it was never designed to support such an interconnected reality.</p>
<p>When it comes to modern devices and cloud services, however, there&#8217;s no such thing as a private API. More to the point, Tesla didn&#8217;t take the most basic precautions to make the API a truly private API, nor did they take the most basic precautions to protect users against the use of the API in third-party applications. In other words, it&#8217;s the worst of all worlds&mdash;not published, private, or secure.</p>
<p>I sincerely believe that ultimately there is no such thing as a private API for consumption over the public Internet. Certainly there exist APIs of all sorts to support private communications between components of an application. The <a href="http://www.enstratius.com">Dell MCM</a> software acquired from the company I founded, Enstratius, has both a public <a href="http://docs.enstratius.com/console/api/api.html">RESTful API</a> and several internal SOAP APIs. We neither publish nor talk much about the internal APIs. What&#8217;s important to note, however, is that a customer can do anything they need to do using the public REST API and we&#8217;ve also taken steps to make sure that customers can&#8217;t reverse engineer and leverage the internal APIs.</p>
<p>While Tesla uses the APIs for communication between their mobile apps and the car, they&#8217;ve neither taken the basic steps to protect it against third-party usage nor is it constructed in a tightly coupled, private fashion. As a consequence, people have reverse engineered the API and begun creating value-added services around it. These aren&#8217;t rogue tools aimed at capturing your credentials&mdash;they are truly valuable services that should not require Tesla&#8217;s blessing. I have even used some of them myself.</p>
<p>The Philips Hue also followed a similar path. The last time I checked, it was also &#8220;not published.&#8221; That didn&#8217;t stop me from writing a generic light bulb management API to talk to the Hue light bulbs and it certainly hasn&#8217;t stopped the rich ecosystem that has grown around those APIs. It also didn&#8217;t stop an outside researcher from <a href="http://www.dhanjani.com/blog/2013/08/hacking-lightbulbs.html">skewering the security model of the Hue API</a>. In short, failure to publish an API doesn&#8217;t make an API a private API. As long as it is consumable over the Internet and in a format that can be reverse engineered, people will reverse engineer it and add value to your creation.</p>
<p>One thing lost in my original article is the fact that I have been praising Tesla for having an API. In fact, in discussing it on Twitter and in other places, I noted that I&#8217;d rather have my Tesla with its flawed API than to have it with no API at all. The key point of the article, however, was that in an interconnected world, it is irresponsible to create an API without protections that enable users to leverage that API with third-party devices and services. It&#8217;s especially irresponsible because common best practices for REST API security (OAuth and request signing) easily deal with that problem without a lot of programming overhead.</p>

            <div id="dsq-comment-message-1796" class="dsq-comment-message"><p>George unfortunately believes that my mail transport agent (Postfix) should use an open web services API with OAUTH to speak to my malware scanner, which should use the same type of API to speak to my virus scanner, which should use the same type of API to speak to my IMAP server, even though it is plainly obvious that those three items are parts of a closed system.</p>
<p>Now, it is a given that someone might want to use my virus scanner to, for example, pipe web pages through&#8230;  or perhaps deliver other messages to my IMAP server.  However, they weren&#8217;t designed for that and it wasn&#8217;t intended, just as Tesla&#8217;s API&#8217;s were intended only for communication between its official mobile apps and its servers.  If they do make their way in that direction, I&#8217;m sure that a new standard or API will emerge, it will be published as an open API, and then George can be happy that his religion once again reigns supreme.</p>
<p>In the author&#8217;s other post, I already have explained the difference between a public REST API and a closed API that just happens to use REST-like syntax.  I already explained that I&#8217;d happily support and upvote every one of George&#8217;s posts IF Tesla had said it was an open API to be used for third parties to provide capabilities to your car. But it&#8217;s not.</p>
<p>Tesla&#8217;s API just doesn&#8217;t fit the author&#8217;s religion, and that&#8217;s ok. He has a right to believe every API should be an open API that uses OAUTH. That&#8217;s okay, in my early days of IT architecture I used to believe everything should be 100% modular and should use very expensive integration capabilites; that compilers shouldn&#8217;t have loop unrolling or use inline code. Someone might use that snippet of code one day and keeping it modular is the only way, etc., etc.</p>
<p>I don&#8217;t believe that anymore. I&#8217;ve moved on. And I can admit that I was wrong back then, that there are places for single-use code, closed API&#8217;s, and narrow optimizations. Moreover, in my daytime job, I am responsible for application integration services and API&#8217;s, and the budget I&#8217;m responsible for doesn&#8217;t leave room for such overly complex architectures.  There are times that tightly-integrated systems are most appropriate, and there are times (such as when an API is published specifically for the purpose of an ecosystem) when the bulletproof capabilities the author desires are required.  Adopting OAUTH, among other requirements needed for open API environments (subscription management, audit, performance management, etc.), does not come free&#8230;  it comes with cost &#8212; cost in design, cost in development, cost in implementation, and cost in operations.  If Tesla isn&#8217;t ready to take that step, then that&#8217;s their prerogative. </p>
<p>When Tesla advertises that API as an open programming interface to its vehicles, then the author is right.  Until then, he espouses an unrealistic ideology.</p>
