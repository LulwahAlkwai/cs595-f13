		<h1 class="heading">Unpatched Mac bug gives attackers &ldquo;super user&rdquo; status by going back in time</h1>
		<h2 class="standalone-deck">Exploiting the five-month-old "sudo" flaw in OS X just got easier.</h2>
		<div class="post-meta">
			  <p class="byline">
  by     <a href="http://arstechnica.com/author/dan-goodin/" rel="author">Dan Goodin</a>
  -    <span class="date" data-time="1377728428">Aug 28, 2013 10:20 pm UTC</span>
  </p>

			<div class="corner-info">

		<div class="article-content clearfix">
    
<figure class="intro-image image center full-width" style="width:640px">

    <figcaption class="caption">
	
			<div class="caption-credit">
							<a rel="nofollow" href="http://www.luispita.com/2008/08/en-pleno-flash-back-veraniego-me-he.html">luispita.com</a>

 <!-- cache hit --><!-- empty --> <p>Researchers have made it easier to exploit a five-month-old security flaw that allows penetration testers and less-ethical hackers to gain nearly unfettered "root" access to Macs over which they already have limited control.</p>
<p>The <a href="http://www.sudo.ws/sudo/alerts/epoch_ticket.html">authentication bypass vulnerability</a> was reported in March and resides in a Unix component known as sudo. While the program is designed to require a password before granting "super user" privileges such as access to other users' files, the bug makes it possible to obtain that sensitive access by resetting the computer clock to January 1, 1970. That date is known in computing circles as the <a href="https://en.wikipedia.org/wiki/Unix_time">Unix epoch</a>, and it represents the beginning of time as measured by the operating system and most of the applications that run on it. By invoking the sudo command and then resetting the date, computers can be <a href="https://cve.mitre.org/cgi-bin/cvename.cgi?name=CVE-2013-1775">tricked into turning over root privileges</a> without a password.</p>
<p>Developers of <a href="http://www.metasploit.com/">Metasploit</a>, an open-source software framework that streamlines the exploitation of vulnerabilities in a wide array of operating systems and applications, <a href="http://packetstormsecurity.com/files/122965">recently added a module</a> that makes it easier to exploit the sudo vulnerability on Macs. The addition capitalizes on the fact that all versions of OS X from 10.7 through the current 10.8.4 remain vulnerable. While the bug also affected many Linux distributions, most of those require a root password to change the computer clock. Macs impose no such restrictions on clock changes, thanks to the <a href="https://developer.apple.com/library/mac/documentation/Darwin/Reference/ManPages/man8/systemsetup.8.html">systemsetup binary</a>.</p>
<p>Mac users should realize that an attacker must satisfy a variety of conditions before being able to exploit this vulnerability. For one, the end-user who is logged in must already have administrator privileges. And for another, the user must have successfully run sudo at least once in the past. And of course, the attacker must already have either physical or remote shell access to the target machine. In other words: this exploit can't be used in the kind of drive-by webpage attacks that last year <a href="http://arstechnica.com/apple/2012/04/flashback-infections-not-waning-after-all-650000-macs-still-hijacked/">infected some 650,000 Macs with the Flashback malware</a>. This doesn't mean it's a non-issue though, since the exploit can be used in concert with other attacks to magnify the damage they can do.</p>
<p>"The bug is significant because it allows any user-level compromise to become root, which in turn exposes things like clear-text passwords from Keychain and makes it possible for the intruder to install a permanent rootkit," HD Moore, the founder of the Metasploit project and the chief research officer at security firm Rapid7, told Ars. "I believe Apple should take this more seriously but am not surprised with the slow response given their history of responding to vulnerabilities in the open source tools they package."</p>
<p>Apple representatives didn't respond to an e-mail seeking comment for this post.</p>
<p><em>Post updated in third-to-last paragraph to changed "invoked" to "successfully run."</em></p>
