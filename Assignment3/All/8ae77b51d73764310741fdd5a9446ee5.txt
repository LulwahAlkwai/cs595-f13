<h3 class='post-title entry-title' itemprop='name'>
PostgreSQL Scalability Analysis Deconstructed
</h3>
<div class='post-header'>
<div class='post-header-line-1'></div>
</div>
<div class='post-body entry-content' id='post-body-5327312516285387277' itemprop='articleBody'>
In 2010, I presented my <a href="http://www.perfdynamics.com/Manifesto/USLscalability.html#tth_sEc1">universal scalability law</a> (<b>USL</b>) at the <a href="http://perfdynamics.blogspot.com/2011/05/quantifying-scalability-ftw-movie.html">SURGE conference</a>. I came away with the impression that nobody really understood what I was talking about (<i>quantifying</i> scalability) or, maybe DevOps types thought it was all too hard (math). Since then, however, I've come to find out that people like Baron Schwartz did get it and have since applied the USL to <a href="http://www.mysqlperformanceblog.com/2011/02/28/is-voltdb-really-as-scalable-as-they-claim/">database scalability analysis</a>. Apparently, things have continued to propagate to the point where others have heard about the USL from Baron and are now using it too. 
<p>
Robert Haas is one of those people and he has applied the USL to <a href="http://rhaas.blogspot.com/2011/09/scalability-in-graphical-form-analyzed.html">Postgres scalability analysis</a>. This is all good news. However, there are plenty of traps for new players and Robert has walked in several of them to the point where, by his own admission, he became confused about what conclusions could be drawn from his USL results. In fact, he analyzed three cases:
<ol>
<li> PostgreSQL 9.1
<li> PostgreSQL 9.2 with fast locking
<li> PostgreSQL 9.2 current release
</ol>
I know nothing about Postgres but thankfully, Robert tabulated on his blog the performance data he used and that allows me to deconstruct what he did with the USL. Here, I am only going to review the first of these cases: PostgreSQL 9.1 scalability. I intend to return to the claimed <i>superlinear effects</i> in another blog post. <a name='more'></a>
<p>
<h4><font color="darkblue">The USL in Brief</font></h4>
The USL is expressed in terms of the <i>relative capacity</i> 
\begin{equation}
C_N = \frac{X_N}{X_1}
\end{equation}
a ratio defined exclusively by the <b>data</b>; the actual throughput <b>measurements</b>, $X_N$. Here, $X_1$ is the throughput measured for a single client $N=1$ load and $X_N$ is the throughput measured with any $N > 1$ clients. 
<p>
But that's only one side of the proverbial coin. On the flip side, the USL provides the following abstract <b>model</b> 
\begin{equation}
C_N = \frac{N}{1 + \alpha N + \beta N (N-1)}
\end{equation}
against which we can regress those same throughput measurements. The USL equation states that the relative capacity is characterized by <b>three</b> effects:
<ol>
<li> The level of <i>concurrency</i> or ideal parallelism (the '1' in the denominator)
<li> Degradation of linear scaling due to <i>contention</i> effects (the $\alpha$ term)
<li> Retrograde scaling due to <i>coherency</i> delays (the $\beta$ term)
</ol>
Since each of these physical effects has a corresponding term in the USL model, the assertion is:  <i>measurements must match the model</i>. Where they don't match, start explaining why not. The method of nonlinear statistical regression is used to determine what values of the $\alpha$ and $\beta$ parameters best account for the variation in the scalability measurements. The relative capacity $C_N$ is a <b>normalized</b> number. To get the corresponding throughput values (that can be compared with measurements), multiply the relative capacity by $X_1$, i.e., $X_N = C_N \, X_1$. More details about regression techniques and the USL are presented in my Guerrilla capacity planning <a href="http://www.perfdynamics.com/iBook/gcap.html">book</a> and <a href="http://www.perfdynamics.com/Classes/Outlines/guerilla.html">training class</a>.
<p>
My purpose here is not to discourage Robert or anyone else from using the USL model, but if you are going to use it, then it really is in your own interest to learn to use it correctly. It's not just a matter of throwing performance data at the USL model and seeing if the plots look like what you were expecting. If the USL always did what we expected, there would be little point in doing it at all because no new insights would be gleaned. It's also not a one-shot, but an <b>exploratory process</b> in which the USL should be applied iteratively. Performance modeling and analysis is a very technical skill, and that's why I run <a href="http://www.perfdynamics.com/Classes/schedule.html">Guerrilla training classes</a>. To give you a better idea of how all that should go, let's see what the USL analysis of Postgres 9.1 looks like when it's done a bit more rigorously.
<p>

<h4><font color="darkblue">The Original Analysis</font></h4>
For reference, the full complement of load test points runs between $N = 1$ and $N = 80$ clients. However, the original plot (Fig. 1 by Robert) only shows the USL analysis performed with <a href="http://www.gnuplot.info/">gnuplot</a> on a <b>subset</b> of data points for $N \le 32$ clients (shown as little circles).
<p>
<div class="separator" style="clear: both; text-align: center;">
<a href="http://3.bp.blogspot.com/-M8c5luF4mUI/T4D8oDFMtjI/AAAAAAAABDs/qDTdXiwqNlg/s1600/gnuplot-pg91vu32.png" imageanchor="1" style="margin-left:1em; margin-right:1em"><img border="0" height="300" width="400" src="http://3.bp.blogspot.com/-M8c5luF4mUI/T4D8oDFMtjI/AAAAAAAABDs/qDTdXiwqNlg/s400/gnuplot-pg91vu32.png" /></a>
<center>Figure 1. Original PG 9.1 scalability analysis in gnuplot for $N \le 32$ clients.</center>
</div>
<p>
What happened to the remaining 48 data points? Apparently, Robert decided that he wanted "to avoid confusing the tool." It's not clear whether that is an anthropomorphic reference to the limitations of <a href="http://www.gnuplot.info/">gnuplot</a> (never used it) or the <a href="http://www.perfdynamics.com/Manifesto/USLscalability.html">USL</a>. Regardless, that statement tells me more about the <i>modeler</i> than the model. Only modelers get confused, not models. How can that happen? Let me count the ways:
<ol>
<li> If all three throughput measurements $X_N$ per each client load point $N$ were shown (rather than the median) there would be a <b>spread</b> in the data. Which is the "true" value?

<li> <a href="http://www.perfdynamics.com/Manifesto/gcaprules.html#tth_sEc1.16"><b>Guerrilla Mantra 1.16</b></a>: Data are not divine.

<li> <a href="http://www.perfdynamics.com/Manifesto/gcaprules.html#tth_sEc2.25"><b>Guerrilla Mantra 2.25</b></a>: All measurements are wrong by definition.

<li> Benchmarks (especially those involving databases) are highly complex <b>simulations</b> that provide a myriad of opportunities for things to go wrong with the measurements.

<li> <a href="http://www.perfdynamics.com/Manifesto/gcaprules.html#tth_sEc1.16"><b>Guerrilla Mantra 1.16</b></a>: Data comes from the Devil, only models come from God.

<li> Applying a performance model is like legally <a href="http://en.wikipedia.org/wiki/Waterboarding"><b>waterboarding</b></a> your data for the truth.

<li> It is <b>not</b> the goal of the USL to <a href="http://perfdynamics.blogspot.com/2011/06/winking-pink-elephant.html">fit an elephant</a>.
</ol>
<p>
Without a full appreciation of these concepts, there is always the tendency to believe the data over the model. It's that <i>bias</i> that led Robert to fit the USL to a subset of his data. Since the USL curve passes through most of those data points, that's better. Right? <a href="http://perfdynamics.blogspot.com/2011/06/winking-pink-elephant.html">Wrong!</a>
<p>

<p>

<p>
The virtue of including the additional reference information is that it allows me to draw the following conclusions. Suppose Robert had not yet measured a client load above $N = 32$ in his benchmark tests. I can see that PG 9.1 scales quite linearly up through the first dozen or so clients, and then begins to fall away from linear due to the nonzero but small value of $\alpha \approx 1\%$. That looks very promising!
<p>
However, something "bad" seems to happen above $N \approx 20$ clients. The USL predicts that scaling (as depicted by the regression curve) will go severely <b>retrograde</b>. Moreover, because the $\alpha$ and $\beta$ parameters in the USL model have distinct <b>physical interpretations</b>, I can say that the retrograde scaling (if it occurs&mdash;remember, we're pretending we don't have the additional data points yet) will not be due to <i>contention</i> (the $\alpha$ term in the USL) but delays caused by some kind of <i>pairwise exchange</i> (the $N^2$ term in the USL) between the clients&mdash;a <i>coherency</i> penalty. Since I'm not a Postgres expert, I have no idea what that could mean, but Robert might. Furthermore, <b>locks</b> are associated with <b>sequencing</b> or <b>serialization</b> (the linear $N$ term in the denominator of the USL) so, any retrograde behavior will be due to something <i>other</i> than locking.
<p>


<h4><font color="darkblue">Complete USL Analysis</font></h4>
Now, let's suppose the additional measurements become available and we fit the USL to all those data. The result is shown in Fig. 3. This is the result that put Robert off. Why? Because he hasn't yet learnt his catechism about models coming from God.
<p>
<div class="separator" style="clear: both; text-align: center;">
<a href="http://4.bp.blogspot.com/-afpwKumqKic/T4D8ELvNBUI/AAAAAAAABDU/kha4K_OGYgo/s1600/Rplot-pg91usl.png" imageanchor="1" style="margin-left:1em; margin-right:1em"><img border="0" height="371" width="400" src="http://4.bp.blogspot.com/-afpwKumqKic/T4D8ELvNBUI/AAAAAAAABDU/kha4K_OGYgo/s400/Rplot-pg91usl.png" /></a>
<center>Figure 3. ULS analysis of the full complement of 21 data points.</center>
</div>
<p>
Robert looks at the USL curve and says "Why?" I look at the USL curve and say "Why not?" The USL predicts that the peak should be closer to $N = 30$, whereas the data peaks <b>prematurely</b> around $N = 20$ (when viewed from the standpoint of the USL). When you think about it, if something is giving you a <b>hint</b> that the scalability could be better than the current data indicates, wouldn't you want to investigate further to see if that performance improvement might be achievable?

Reviewing Fig. 3 more carefully, I can also say with some authority that PG 9.1 is now scaling better than expected (based on the USL curve in Fig. 2) because it did not go as retrograde as anticipated by the USL, based on the load points up to $N = 32$. In other words, if we already see one improvement, could there be yet more improvements?
<p>
Another way of stating this conclusion is to note the throughput data are somewhat <i>exceeding</i> USL expectations in the window just prior to the predicted USL peak (say, $15 < N < 20$), they are <i>failing</i> to meet USL expectations in the window $20 < N < 60$, but finally recover again in the window $60 < N < 80$. This view of the data immediately raises questions about possible room for performance improvements. Can PG do better? Can the scaling be more consistent? If not, why not? Maybe PG can't get there, but we need to understand why not. In this sense, USL modeling is often more about driving <b>explanations</b> than fitting data.
<p>


<h4><font color="darkblue">Extended USL Analysis</font></h4>
Finally, let's switch gears again and ask, What would happen if we could magically "turn off" the coherency penalty (i.e., set $\beta = 0$) in Fig. 3? The USL prediction is shown in Fig. 4.
<p>
<div class="separator" style="clear: both; text-align: center;">
<a href="http://4.bp.blogspot.com/-rqUsThTxpK8/T4D8MLkFDEI/AAAAAAAABDg/ltDiu6l_xf4/s1600/Rplot-pg91amdOR.png" imageanchor="1" style="margin-left:1em; margin-right:1em"><img border="0" height="371" width="400" src="http://4.bp.blogspot.com/-rqUsThTxpK8/T4D8MLkFDEI/AAAAAAAABDg/ltDiu6l_xf4/s400/Rplot-pg91amdOR.png" /></a>
<center>Figure 4. ULS analysis with $\beta$ parameters suppressed.</center></div>
<p>
This USL analysis says the following. If you could maintain the level of contention at the value in Fig. 3 (i.e. $\alpha \approx 4\%$), then near linear scaling could take throughput performance up to a higher plateau (denoted $X_{roof}$ in the legend) given by: 
\begin{equation}
\frac{X_1}{\alpha} = \frac{4373.30}{0.03855} = 113,444.90~\text{TPS} 
\end{equation}
If we consider the data circles beyond say, $N = 40$ clients in  Fig. 4 to constitute an approximate throughput plateau for PG 9.1, then the predicted $X_{roof}$ plateau would be about <b>4x</b> higher than that. As you can see, under these assumptions, the USL scalability curve would be off the  chart!
<p>
In case you think that's a completely crazy projection coming from the USL, then <a href="http://rhaas.blogspot.com/2011/09/scalability-in-graphical-form-analyzed.html">check out</a> what actually happened to the newer version of Postgres with the so-called <i>fast locking</i> modification. Those PG 9.2 throughput measurements, <a href="http://2.bp.blogspot.com/-2Ft63JuwHO8/ToXX4zw2X8I/AAAAAAAAAEs/l3i3ESLPhWU/s1600/read-scaling.png">shown as green joined data points</a>, reached an $X_{roof}$ that is <b>5x</b> higher than expected from the PG 9.1 data. The PG 9.2 fast lock data makes the PG 9.1 USL prediction look <i>conservative</i>. Some of the additional boost to $X_{roof}$ may have come from the apparent superlinear effect, but I won't go into that aspect here.
<p>
Of course, the architectural changes to Postgres 9.2 were not made as a consequence of any USL modeling (that I know about), but they could have been. I hope you can see now how the <i>modeling scenarios</i> (different parameter values) that I've presented here could have been applied to the PG data within the context of the USL. Most importantly, each of these USL scenarios in Figs. 2, 3 and 4 should be seen as fitting together logically and <i>reinforcing</i> one  another. All that <b>information</b> was hidden in the original data.
<p>
Indeed, I can say with hindsight, that whatever changes were made between PG 9.1 and 9.2 with fast locking, it had the predicted effect of making the USL $\beta$ parameter vanishing small (but not zero since some slight retrograde behavior persists). In addition, the fast locks reduced the $\alpha$ parameter to something smaller than the PG 9.1 value of $\alpha = 0.03855$. Simply put, if the USL had been used to reach these conclusions, I would've been in a position to say, "I told you so." ;-)
<p>
<h4><font color="darkblue">Summary</font></h4>
These are just some of the insights you can gain from the appropriate use of the USL model. In fact, this PostgreSQL example would make a great case study for a <a href="http://www.perfdynamics.com/Classes/schedule.html">Guerrilla training class</a>.

<p>
The reason I ignored the points where N&gt;32 is because the data were collected on a system with 32 cores.  So there are multiple factors limiting scalability here.  Where clients &lt;= cores, we have one set of bottlenecks, principally due to lock contention within PostgreSQL but perhaps also partly due to operating system or hardware effects.  However, once the number of clients exceeds the number of cores, we&#39;re bound to hit a wall: if all the available CPU resources are already in use, adding more clients can&#39;t really continue to improve throughput.  What I want to measure is whether it&#39;s possible to add throughput by adding more hardware resources (cores), NOT whether or not throughput will flatten out when we run out of cores.  The answer to the latter question seems pretty self-evident: if we&#39;re efficiently using every core, then the best we can hope for when we run out of cores is that throughput will remain stable.  In reality, of course, it will drop off slightly, because task-switching is not perfectly efficient.
</p>
